## 1. 이 문서는 무엇인가요?

이 문서는 개발자가 아닌 분들도 서비스의 전체 화면 구성을 쉽게 이해할 수 있도록 정리한 `공유용 화면 설명서`입니다.

쉽게 말해,

- 이 서비스에 어떤 화면이 있고
- 각 화면은 누구를 위한 것이며
- 그 화면에서 사용자가 무엇을 하게 되는지

를 한눈에 이해할 수 있도록 만든 문서입니다.

---

## 2. 이 서비스는 어떤 서비스인가요?

이 서비스는 식품 제조를 맡기고 싶은 회사와, 실제로 제조를 할 수 있는 공급업체를 연결하는 서비스입니다.

단순히 업체를 소개하는 데서 끝나는 것이 아니라,

1. 공급자를 찾고
2. 제조 의뢰를 등록하고
3. 견적을 받고 비교하고
4. 메시지로 상담하고
5. 필요할 때만 연락처를 공유하는

안전한 연결 흐름을 만드는 것이 핵심입니다.

---

## 3. 누가 사용하는 서비스인가요?

### 3.1 요청자

제품 생산을 맡기고 싶은 회사입니다.

주요 행동:

- 사업자 정보 제출
- 공급자 찾기
- 의뢰 등록
- 견적 비교
- 공급자와 상담

### 3.2 공급자

실제로 제품을 제조할 수 있는 업체입니다.

주요 행동:

- 회사/설비 정보 등록
- 검증 서류 제출
- 의뢰 확인
- 견적 제출
- 요청자와 상담

### 3.3 관리자

서비스 운영자입니다.

주요 행동:

- 제출 서류 검토
- 승인 / 보류 / 반려 처리
- 공지 작성
- 운영 통계 확인

### 3.4 일반 방문자

회원가입하지 않아도 일부 공개 화면을 볼 수 있는 사용자입니다.

주요 행동:

- 공지사항 보기
- 일부 공개 공급자 정보 보기

---

## 4. 전체 화면 구성 한눈에 보기

이 서비스의 화면은 크게 다섯 덩어리로 나눌 수 있습니다.

1. 로그인/회원가입 같은 공통 화면
2. 요청자용 화면
3. 공급자용 화면
4. 요청자와 공급자가 대화하는 메시지 화면
5. 관리자 운영 화면

---

## 5. 요청자는 어떤 흐름으로 사용하나요?

요청자의 기본 흐름은 아래와 같습니다.

1. 회원가입 / 로그인
2. 사업자 정보 제출
3. 승인 상태 확인
4. 공급자 탐색
5. 공급자 상세 확인
6. 의뢰 등록
7. 내 의뢰 확인
8. 견적 비교 및 선택
9. 메시지 상담
10. 필요 시 연락처 공유

이 흐름에서 중요한 점은, 요청자는 `사업자 승인 전에는 실제 의뢰 등록이 제한된다`는 것입니다.

즉, 먼저 신뢰를 확인한 뒤 실제 제조 문의를 진행하는 구조입니다.

---

## 6. 공급자는 어떤 흐름으로 사용하나요?

공급자의 기본 흐름은 아래와 같습니다.

1. 회원가입 / 로그인
2. 공급자 프로필 작성
3. 검증 서류 제출
4. 검증 결과 확인
5. 의뢰 목록 확인
6. 의뢰 상세 확인
7. 견적 제출
8. 내 견적 관리
9. 메시지 상담
10. 필요 시 연락처 공유 승인

즉, 공급자는 먼저 “신뢰할 수 있는 제조업체인지”를 보여주는 과정이 있고, 그 이후에 의뢰를 받고 견적을 낼 수 있습니다.

---

## 7. 관리자는 어떤 흐름으로 사용하나요?

관리자의 기본 흐름은 아래와 같습니다.

1. 관리자 로그인
2. 검수 큐 확인
3. 제출 자료 확인
4. 승인 / 보류 / 반려 처리
5. 공지사항 관리
6. 운영 통계 확인

즉, 관리자는 이 서비스에서 `신뢰를 유지하는 역할`과 `운영 안내를 담당하는 역할`을 동시에 맡습니다.

---

## 8. 주요 화면 설명

아래는 실제 운영과 사용성 측면에서 중요한 화면들만 쉽게 풀어쓴 설명입니다.

### 8.1 로그인 화면

- 누구를 위한 화면인가: 전체 사용자
- 왜 필요한가: 서비스에 들어오는 첫 관문
- 무엇을 하는가: 이메일과 비밀번호를 입력해 로그인
- 중요한 포인트: 로그인 실패 시 바로 이해할 수 있는 오류 안내가 필요

### 8.2 회원가입 화면

- 누구를 위한 화면인가: 요청자, 공급자
- 왜 필요한가: 서비스 첫 진입 화면
- 무엇을 하는가: 이메일, 비밀번호, 업체명, 역할을 입력해 계정 생성
- 중요한 포인트: “나는 요청자인지 공급자인지”를 초기에 분명히 나누는 것이 중요

### 8.3 사업자 정보 등록 화면

- 누구를 위한 화면인가: 요청자
- 왜 필요한가: 승인된 요청자만 실제 의뢰를 올릴 수 있도록 하기 위해
- 무엇을 하는가: 사업자 정보와 담당자 정보를 입력하고 제출
- 사용자가 기대하는 것:
  - 지금 내가 승인 전인지, 대기 중인지, 승인 완료인지 바로 알 수 있어야 함
  - 승인 완료 후 바로 의뢰 등록으로 이어질 수 있어야 함

### 8.4 공급자 탐색 화면

- 누구를 위한 화면인가: 요청자
- 왜 필요한가: 적합한 제조업체를 찾는 핵심 화면
- 무엇을 하는가:
  - 회사명 검색
  - 지역 / 카테고리 / 생산량 / MOQ / OEM/ODM 조건으로 필터
  - 공급자 상세 화면 이동
- 중요한 포인트:
  - 복잡해 보여도 안 되고, 너무 정보가 부족해도 안 됨
  - 한 번에 “이 업체가 우리 조건에 맞는지”를 빨리 판단할 수 있어야 함

### 8.5 공급자 상세 화면

- 누구를 위한 화면인가: 요청자
- 왜 필요한가: 특정 공급자의 신뢰도와 제조 가능성을 자세히 판단하기 위해
- 무엇을 보는가:
  - 회사 기본 정보
  - 제조 역량
  - 인증서
  - 포트폴리오
  - 소개 문구
- 무엇을 하는가:
  - “이 공급자에게 의뢰하기” 버튼으로 바로 의뢰 연결

### 8.6 의뢰 등록 화면

- 누구를 위한 화면인가: 요청자
- 왜 필요한가: 제조 요청 조건을 구조적으로 정리하기 위해
- 무엇을 입력하는가:
  - 제품/카테고리
  - 희망 수량
  - 희망 단가
  - 인증 조건
  - 원재료 / 포장 / 납기 / 비고
  - 공개형인지, 특정 업체 지정형인지
- 중요한 포인트:
  - 입력이 너무 어려우면 실제 등록 전환이 떨어짐
  - 필요한 항목은 구조화하되, 사용자가 문장을 쓰는 부담은 줄여야 함

### 8.7 내 의뢰 목록 / 의뢰 상세 화면

- 누구를 위한 화면인가: 요청자
- 왜 필요한가: 등록한 의뢰를 관리하기 위해
- 무엇을 할 수 있는가:
  - 수정
  - 게시
  - 마감
  - 취소
  - 견적 비교 이동
  - 지정 공급자와 대화 시작
- 중요한 포인트:
  - 상태에 따라 버튼이 달라져야 함
  - 이미 마감된 건 실수로 다시 수정되지 않도록 읽기 중심으로 보여야 함

### 8.8 견적 비교 화면

- 누구를 위한 화면인가: 요청자
- 왜 필요한가: 여러 공급자의 견적을 한 자리에서 비교하기 위해
- 무엇을 보는가:
  - 공급자명
  - 제출일
  - 단가
  - MOQ
  - 납기
  - 샘플 비용
- 무엇을 하는가:
  - 상세 보기
  - 메시지 이동
  - 견적 선택
  - 견적 거절
- 중요한 포인트:
  - 이 화면은 “비교”가 목적이므로 표 형태가 잘 맞음
  - 선택 시 다른 견적 처리 방식이 분명히 안내돼야 함

### 8.9 공급자 프로필 관리 화면

- 누구를 위한 화면인가: 공급자
- 왜 필요한가: 공급자가 스스로를 소개하는 핵심 화면이기 때문
- 무엇을 입력하는가:
  - 회사명, 대표자명, 연락처, 지역
  - 카테고리, 설비, 생산량, MOQ
  - OEM/ODM 가능 여부
  - 소개 문구
- 중요한 포인트:
  - 이 화면은 단순 정보 입력 화면이 아니라 “영업/신뢰 화면”이기도 함
  - 요청자가 봤을 때 이해하기 쉬운 정보 구조여야 함

### 8.10 공급자 의뢰 상세 / 견적 제출 화면

- 누구를 위한 화면인가: 공급자
- 왜 필요한가: 공급자가 실제로 견적을 낼 수 있어야 하기 때문
- 무엇을 하는가:
  - 의뢰 조건 확인
  - 예상 단가, MOQ, 납기, 샘플 비용 입력
  - 제출 전 요약 확인
- 중요한 포인트:
  - 잘못 제출하면 거래에 영향을 주기 때문에, 제출 전 확인 단계가 필요함

### 8.11 내 견적 관리 화면

- 누구를 위한 화면인가: 공급자
- 왜 필요한가: 제출한 견적을 다시 확인하고 관리하기 위해
- 무엇을 하는가:
  - 수정
  - 철회
  - 메시지 이동
- 중요한 포인트:
  - 제출 완료 상태에서만 수정/철회 가능하게 제한하는 것이 혼선을 줄임

### 8.12 메시지 목록 / 대화 상세 화면

- 누구를 위한 화면인가: 요청자 / 공급자
- 왜 필요한가: 의뢰와 견적 이후 실제 상담이 이어지는 핵심 공간이기 때문
- 무엇을 하는가:
  - 메시지 목록에서 대화 선택
  - 텍스트/파일 전송
  - 상담 이어가기
  - 연락처 공유 요청 또는 승인
- 중요한 포인트:
  - 연락처는 바로 보이면 안 되고, 양쪽 동의 후에만 공개되어야 함
  - 읽지 않은 메시지는 눈에 잘 띄어야 함

### 8.13 공지사항 관리 화면

- 누구를 위한 화면인가: 관리자
- 왜 필요한가: 운영 공지를 게시하기 위해
- 무엇을 하는가:
  - 새 공지 작성
  - 수정
  - 게시
  - 보관
- 중요한 포인트:
  - 관리자는 빠르게 작성/상태 변경이 가능해야 함
  - 일반 사용자는 게시된 공지만 보게 해야 함

### 8.14 통계 대시보드 화면

- 누구를 위한 화면인가: 관리자
- 왜 필요한가: 운영 현황을 숫자로 빠르게 파악하기 위해
- 무엇을 보는가:
  - 전체 사용자 수
  - 요청자/공급자/관리자 수
  - 공급자 상태별 분포
  - 검수 현황
  - 의뢰 현황
- 중요한 포인트:
  - 숫자를 단순히 나열하지 않고, 한눈에 비교되는 구조가 중요함

## 9. 운영 관점에서 꼭 봐야 할 포인트

이 서비스는 결국 아래 세 가지가 잘 보이는 화면 구조여야 합니다.

### 9.1 신뢰가 보이는가?

- 공급자 검증 상태가 잘 보이는가
- 사업자 승인 전 제한이 잘 보이는가
- 연락처는 정말 필요한 순간에만 보이는가

### 9.2 비교가 쉬운가?

- 공급자를 찾고 고르는 과정이 어렵지 않은가
- 견적 비교가 한눈에 되는가
- 메시지 상담으로 자연스럽게 이어지는가

### 9.3 운영이 쉬운가?

- 관리자가 검수를 빠르게 처리할 수 있는가
- 공지와 통계를 쉽게 다룰 수 있는가
- 문제가 생겼을 때 어떤 상태인지 바로 파악할 수 있는가

---

## 10. 현재 기준 확인이 필요한 부분

현재 PRD 기준으로 보면, 아래는 별도 확인이 필요합니다.

### 관리자 회원 목록 화면

- 문서상 필요성이 언급돼 있지만
- 현재는 검수 화면, 공지 화면, 통계 화면 중심으로 구현되어 있음

즉, 아래 둘 중 하나로 정리해야 합니다.

1. 이미 다른 화면에서 사실상 해결되는 요구인지 확인
2. 별도 관리자 회원 목록 화면이 필요한지 다음 단계 과업으로 확정

---

## 11. 한 줄 정리

이 서비스의 화면은 “좋은 공급자를 찾고, 제조를 요청하고, 견적을 비교하고, 안전하게 상담한 뒤 거래 가능성을 높이는 과정”을 자연스럽게 이어주는 구조여야 합니다.

관리자 화면은 이 흐름이 신뢰 있게 돌아가도록 뒤에서 검수하고 안내하고 확인하는 운영 도구 역할을 합니다.
