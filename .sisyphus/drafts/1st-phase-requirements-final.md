## 1. 문서 목적

이 문서는 사용자 답변과 기존 PRD 초안을 다시 정리해 **실제로 개발 가능한 MVP 기준선**을 정의한다.

이 문서에서 가장 중요한 원칙은 아래 두 가지다.

1. 장기 비전과 MVP 범위를 분리한다.
2. 후속 PRD와 요약 문서는 이 문서를 넘어서는 약속을 하지 않는다.

---

## 2. 제품 방향

### 2.1 장기 비전

장기적으로는 아래를 포함하는 식품 제조 B2B 플랫폼을 지향한다.

- 제조사 탐색 및 매칭
- 신뢰/검증 관리
- 거래 전 커뮤니케이션 지원
- 콘텐츠, 커뮤니티, 교육/컨설팅
- 이후 AI 및 외부 연동 기능

### 2.2 Phase 1 / MVP 목표

Phase 1의 목표는 `풀 패키지 완성`이 아니라 아래의 **신뢰형 매칭 코어**를 만드는 것이다.

- 검증 가능한 공급자 프로필
- 요청자 의뢰 등록
- 견적 제출 및 비교 흐름
- 메시지 기반 상담
- 관리자 검수 및 운영 기본 기능

### 2.3 Phase 1에서 제외하는 항목

아래 항목은 중요하지만 MVP 필수 범위로 두지 않는다.

- 커뮤니티 게시판 확장
- 교육/컨설팅 운영 기능
- 리뷰/평점 시스템
- 지도 기반 탐색
- WebSocket 기반 실시간성 강화
- OCR/외부 검증 API 자동화
- AI 추천, 챗봇, 가격 예측
- 결제, 전자계약, 에스크로

## 3. 사용자 및 권한 기준

### 3.1 주요 사용자

| 사용자 | 설명 | MVP 참여 방식 |
|--------|------|---------------|
| 요청자 | 제조를 의뢰하는 사업자 기반 브랜드/운영 조직 | 사업자 정보 제출 후 의뢰 가능 |
| 공급자 | 식품 제조 역량을 가진 공급자/제조사 | 가입 가능, 검증 상태에 따라 노출/권한 차등 |
| 관리자 | 검수 및 운영 담당 | 인증 심사, 공지, 통계 관리 |

### 3.2 인증/검증 정책

| 항목 | MVP 기준 |
|------|----------|
| 요청자 가입 | 사업자 기반 가입을 기본으로 함 |
| 요청자 의뢰 권한 | 사업자 승인 후에만 의뢰 등록/수정 가능 |
| 공급자 가입 | 가입 자체는 가능 |
| 공급자 활동 권한 | 검증 상태에 따라 제한/확장 |
| 사업자 검증 | MVP에서는 수동 심사 중심 |
| HACCP/기타 인증 | 정보 입력 + 문서 업로드 + 관리자 검토 |
| 신뢰 표시 | 검증된 공급자 배지/상태 표시 |

정리하면, 공급자는 `가입 허용`, `검증 기반 신뢰/노출 차등`이 맞는 기준이다. 기존 문서의 `공급자 모두 HACCP 필수` 같은 단정 표현은 이 문서 기준으로 정리한다.

## 4. Phase 1 / MVP 포함 범위

### 4.1 회원 및 인증

- 이메일 기반 가입/로그인
- 역할 선택: 요청자 / 공급자 / 관리자
- 요청자 사업자 정보 제출
- 공급자 사업자 정보 및 인증 정보 제출
- 관리자 수동 검수
- 검증 상태 표시
- 요청자 사업자 승인 후 의뢰 등록 가능

### 4.2 공급자 프로필

#### 필수 항목

- 회사명
- 대표자명
- 위치
- 연락처
- 취급 카테고리
- 보유 설비
- 월 생산 가능량
- MOQ
- OEM/ODM 가능 여부
- 원재료 제공 여부
- 포장/라벨링 가능 여부

#### 선택 또는 검수 항목

- HACCP 여부 및 문서
- ISO/FSSC 등 추가 인증
- 포트폴리오 이미지
- 주요 거래처
- 소개 문구

### 4.3 검색 및 탐색

#### MVP 필수 기능

- 키워드 검색
- 카드 뷰
- 리스트 뷰
- 카테고리 필터
- 지역 필터
- 인증 상태 필터
- 생산 역량 필터
- MOQ 필터
- OEM/ODM 필터

#### MVP에서 보류하는 기능

- 리뷰 기반 정렬/필터
- 지도 기반 탐색
- 거리 기반 정렬

### 4.4 의뢰 및 견적

- 요청자 의뢰 등록
- 공개 의뢰
- 선택 공급자 대상 문의
- 공급자 견적 제출
- 견적 비교
- 연락처 공유 동의 흐름

추가 운영 규칙:

- 요청자는 사업자 승인 후에만 의뢰를 등록할 수 있다.
- 견적 수정은 `submitted` 상태에서만 가능하며 수정 이력을 남긴다.

견적 정보는 아래를 기본 구조로 둔다.

- 예상 단가
- MOQ
- 납기
- 샘플 비용
- 비고

### 4.5 커뮤니케이션

Phase 1의 기준 용어는 `실시간 채팅`보다 `메시지 스레드`가 더 정확하다.

- 텍스트 메시지
- 이미지/파일 첨부
- 읽음 상태
- 연락처 공유 동의 후 외부 연락처 표시
- 웹 내 알림 또는 새로고침 기반 확인

추가 운영 규칙:

- 메시지 스레드는 첫 견적 제출 시 자동 생성되거나, 요청자가 `상담 시작`으로 수동 생성할 수 있다.
- 같은 요청자-공급자-의뢰 조합에는 활성 스레드 1개만 허용한다.
- 연락처 공유 요청은 철회 가능하며, 철회 후 다시 요청할 수 있다.

Phase 1에서는 WebSocket, 타이핑 인디케이터, 푸시 알림을 전제하지 않는다.

### 4.6 관리자 기능

- 회원 목록 조회
- 사업자/인증 검수
- 공급자 상태 변경
- 공지사항 등록
- 핵심 통계 확인

추가 운영 규칙:

- `hold`는 보완 요청 상태다.
- `reject`는 현재 제출 건 반려 상태다.
- `hold`와 `reject` 모두 이후 재제출이 가능하다.

## 6. UX/UI 기준

### 6.1 메인 화면 우선순위

1. 검색 진입
2. 카테고리 탐색
3. 추천 또는 검증된 공급자 노출
4. 공지/운영 메시지
5. 확장 영역 배너

### 6.2 시각 방향

- 전문적이어야 한다.
- 동시에 처음 보는 사용자도 이해하기 쉬워야 한다.
- 식품 제조 용어는 설명 가능한 UI가 필요하다.

### 6.3 모바일 대응

MVP는 반응형 웹을 기본으로 한다. 다만 `모바일에서도 모든 기능을 완벽히 동일하게 사용`을 문서 약속으로 과하게 쓰지 않고, 핵심 흐름 우선 최적화로 표현한다.

## 7. 기술 기준

### 7.1 현재 기준 스택

| 영역 | 기준 |
|------|------|
| 프론트엔드 | React |
| 백엔드 | Kotlin Spring Boot |
| 관계형 저장소 | MariaDB 또는 MySQL 호환 계열 |
| 문서/조회 저장소 | MongoDB |
| 설계 방향 | CQRS |

### 7.2 아직 확정하지 않는 항목

- 인프라 사업자
- 파일 저장소 세부 구현
- CDN
- 실시간 통신 방식
- 지도 제공자
- 외부 인증 API 제공자

이 항목들은 후속 설계 문서에서 결정한다. 요구사항 문서 단계에서 구현 세부를 확정된 것처럼 쓰지 않는다.

## 8. 일정과 완료 기준

기존 문서의 `1-8주`, `1-3개월`, `6개월 내`, `모든 기능 완성 후 런칭`은 서로 충돌하므로, 이 문서에서는 이를 고정 일정으로 쓰지 않는다.

대신 아래 기준을 사용한다.

### 8.1 Phase 1 완료 조건

- 요청자 가입/의뢰 등록 가능
- 공급자 가입/프로필/검증 상태 관리 가능
- 견적 제출/비교 가능
- 메시지 스레드 및 파일 첨부 가능
- 관리자 검수 및 공지 기능 가능
- 핵심 역할별 흐름 테스트 통과

### 8.2 런칭 기준

`모든 미래 기능 완성`이 아니라 `Phase 1 완료 조건 충족 + 운영 가능 상태`를 런칭 기준으로 본다.
