## 1. 문서 목적

이 문서는 Phase 1 / MVP에서 실제로 구현해야 할 범위를 구현 관점으로 정리한 문서다.

이 문서는 장기 비전을 설명하되, 구현 범위는 반드시 MVP 기준선에 맞춘다.

---

## 2. 제품 프레이밍

### 2.1 장기 비전

장기적으로는 식품 제조 매칭, 검증, 운영 지원, 커뮤니티/콘텐츠, AI 기능까지 확장 가능한 플랫폼을 지향한다.

### 2.2 v1.0 MVP 정의

v1.0 MVP는 `신뢰형 매칭 코어`다.

포함 범위:

- 사업자 기반 사용자 흐름
- 공급자 검증 상태 관리
- 공급자 프로필
- 검색/필터
- 의뢰/견적 흐름
- 메시지 스레드
- 관리자 검수 기본 기능

제외 범위:

- 커뮤니티 확장
- 교육/컨설팅 운영 기능
- 리뷰/평점
- 지도 기반 탐색
- WebSocket 기반 실시간성 강화
- OCR/외부 검증 API 자동화
- AI 추천/챗봇/가격 예측
- 결제/에스크로/전자계약

## 3. 사용자 역할

| 역할 | 설명 | MVP 핵심 행위 |
|------|------|---------------|
| 요청자 | 제조를 의뢰하는 사업자 기반 사용자 | 가입, 의뢰 등록, 공급자 탐색, 견적 비교, 메시지 상담 |
| 공급자 | 제조 역량을 가진 업체 | 가입, 프로필 작성, 검증 서류 제출, 견적 제출, 메시지 응답 |
| 관리자 | 플랫폼 운영자 | 인증 심사, 상태 변경, 공지 작성, 기초 통계 확인 |

## 4. Phase 1 핵심 기능

### 4.1 AUTH-001 회원가입 및 검증 상태 관리

#### 설명

요청자와 공급자는 모두 가입할 수 있어야 하며, 공급자는 검증 상태에 따라 기능 권한과 노출 수준이 달라져야 한다.

#### 수용 기준

1. 이메일 기반 가입/로그인이 가능하다.
2. 요청자/공급자 역할을 선택할 수 있다.
3. 요청자는 사업자 정보를 제출할 수 있다.
4. 공급자는 사업자 정보와 인증 정보를 제출할 수 있다.
5. 관리자는 제출 정보를 검수하고 상태를 변경할 수 있다.
6. 공급자 프로필에는 검증 상태가 표시된다.

#### 비고

- MVP에서는 수동 심사를 기본으로 한다.
- 외부 사업자 검증 API는 필수 의존성이 아니다.

---

### 4.2 PROFILE-001 공급자 프로필

#### 필수 항목

- 회사명
- 대표자명
- 위치
- 연락처
- 카테고리
- 보유 설비
- 월 생산 가능량
- MOQ
- OEM/ODM 가능 여부
- 원재료 제공 여부
- 포장/라벨링 가능 여부

#### 선택/검수 항목

- HACCP 여부 및 문서
- ISO/FSSC 등 추가 인증
- 포트폴리오 이미지
- 주요 거래처
- 회사 소개

#### 수용 기준

1. 공급자는 프로필을 작성/수정할 수 있다.
2. 요청자는 프로필에서 핵심 생산 역량과 검증 상태를 확인할 수 있다.
3. 인증 정보는 구조화된 필드로 표시된다.
4. 관리자 검수 결과가 프로필 노출 상태에 반영된다.

---

### 4.3 SEARCH-001 검색 및 필터링

#### MVP 필수 기능

- 키워드 검색
- 카드 뷰
- 리스트 뷰
- 카테고리 필터
- 지역 필터
- 인증 상태 필터
- 생산 역량 필터
- MOQ 필터
- OEM/ODM 필터

#### 수용 기준

1. 요청자는 공급자 목록을 검색할 수 있다.
2. 필터 조합에 따라 결과를 좁힐 수 있다.
3. 카드/리스트 뷰 전환이 가능하다.
4. 공급자 카드에는 최소한 회사명, 카테고리, 지역, 검증 상태, 주요 역량이 보인다.

#### Phase 1 제외

- 리뷰 정렬/필터
- 거리 기반 정렬
- 지도 뷰

---

### 4.4 REQUEST-001 의뢰 등록

#### 기본 구조

- 제품명
- 카테고리
- 희망 생산량
- 희망 단가 범위
- 요구 인증 여부
- 원재료 제공 여부
- 포장/라벨링 요구
- 납기 요구
- 첨부 자료
- 비고

#### 수용 기준

1. 요청자는 의뢰를 생성할 수 있다.
2. 의뢰는 공개형 또는 선택 공급자 대상 문의형으로 운영할 수 있다.
3. 공급자는 관련 의뢰를 보고 응답할 수 있다.

---

### 4.5 QUOTE-001 견적 제출 및 비교

#### 견적 항목

- 예상 단가
- MOQ
- 납기
- 샘플 비용
- 추가 메모

#### 수용 기준

1. 공급자는 의뢰에 대해 견적을 제출할 수 있다.
2. 요청자는 받은 견적을 비교할 수 있다.
3. 견적 비교 후 특정 공급자와 후속 상담을 이어갈 수 있다.

---

### 4.6 MESSAGE-001 메시지 스레드

#### 설명

Phase 1의 커뮤니케이션은 `실시간 채팅`보다 `메시지 스레드`로 정의한다.

#### 수용 기준

1. 요청자와 공급자는 의뢰/견적 맥락에서 메시지를 주고받을 수 있다.
2. 이미지와 파일 첨부가 가능하다.
3. 읽음 상태를 표시할 수 있다.
4. 연락처는 상호 동의 후 공개할 수 있다.
5. 사용자는 웹 내에서 새 메시지를 확인할 수 있다.

#### Phase 1 제외

- WebSocket 기반 실시간 전송 보장
- 푸시 알림
- 타이핑 인디케이터
- 음성/영상 통화

---

### 4.7 ADMIN-001 관리자 기본 기능

#### 수용 기준

1. 관리자는 사용자/공급자 목록을 조회할 수 있다.
2. 사업자/인증 제출 건을 심사할 수 있다.
3. 공급자의 상태를 변경할 수 있다.
4. 공지사항을 등록할 수 있다.
5. 기본 통계 지표를 확인할 수 있다.

## 6. 비기능 요구사항

### 6.1 사용성

- 검색 진입이 쉬워야 한다.
- 식품 제조 용어는 설명 가능한 UI를 제공해야 한다.
- 관리자 검수 상태가 사용자에게 명확히 보여야 한다.

### 6.2 반응형

- 데스크톱 우선으로 설계하되, 핵심 사용자 흐름은 모바일에서도 수행 가능해야 한다.

### 6.3 보안

- 인증/검증 문서는 접근 제어가 되어야 한다.
- 민감 정보는 권한에 따라 노출을 제한해야 한다.
- 파일 업로드는 형식 제한과 기본 검증이 필요하다.

## 8. 외부 의존성 원칙

Phase 1은 외부 서비스 의존 없이도 핵심 흐름이 성립해야 한다.

따라서 아래 항목은 `있으면 좋은 연동`이지 `MVP 선행조건`이 아니다.

- 지도 API
- 사업자 검증 API
- OCR 서비스
- 결제 서비스

## 9. 검증 시나리오

### 요청자 시나리오

1. 요청자가 가입한다.
2. 사업자 정보를 제출한다.
3. 공급자를 탐색한다.
4. 의뢰를 등록한다.
5. 견적을 비교한다.
6. 특정 공급자와 메시지 상담을 이어간다.

### 공급자 시나리오

1. 공급자가 가입한다.
2. 프로필과 검증 서류를 제출한다.
3. 관리자가 상태를 검수한다.
4. 공급자가 의뢰에 견적을 제출한다.
5. 요청자와 메시지를 주고받는다.

### 관리자 시나리오

1. 제출 서류를 확인한다.
2. 상태를 승인/보류/반려로 변경한다.
3. 공지사항을 게시한다.
4. 기초 운영 지표를 확인한다.
