## 1. 문서 목적

이 문서는 v1.0 MVP를 기준으로 서비스에 필요한 화면을 비개발자도 이해할 수 있는 방식으로 정리한 화면 기획서다.

기술 구현 방식보다는 아래 질문에 답하도록 작성한다.

- 이 화면은 왜 필요한가?
- 누가 보는가?
- 이 화면에서 무엇을 확인하고 무엇을 행동하는가?
- 어떤 상태에서 어떤 문구와 버튼이 보여야 하는가?

### 어려운 표현 쉽게 보기

- `MVP`: 이번 1차 개발에서 꼭 필요한 최소 기능 묶음
- `PRD`: 무엇을 만들어야 하는지 정리한 기준 문서
- `UX`: 사용자가 쓰면서 느끼는 편의성과 이해하기 쉬운 정도
- `UI`: 실제 화면의 구성, 버튼, 입력칸, 표 같은 보이는 요소
- `상태 기획`: 어떤 상황에서 어떤 버튼과 문구가 보여야 하는지 정하는 것
- `completion debt`: 원래 이번 단계에서 닫혔어야 하지만 아직 덜 마무리된 항목

## 2. 서비스 한 줄 설명

이 서비스는 식품 제조를 맡기고 싶은 요청자와 제조 역량을 가진 공급자를 연결하는 `신뢰형 매칭 플랫폼`이다.

핵심은 단순한 소개가 아니라,

- 검증된 공급자 정보를 보고,
- 의뢰를 등록하고,
- 견적을 비교하고,
- 메시지로 상담을 이어가며,
- 필요할 때만 연락처를 공유하는 흐름을 만드는 것이다.

---

## 3. 주요 사용자

### 3.1 요청자

- 제품 제조를 맡기고 싶은 사업자
- 사업자 승인 후 의뢰 등록 가능
- 공급자 탐색, 의뢰 등록, 견적 비교, 상담 진행이 핵심

### 3.2 공급자

- 제조를 수행하는 업체
- 프로필 작성과 검증 제출이 우선
- 의뢰 확인, 견적 제출, 상담 응답이 핵심

### 3.3 관리자

- 제출 서류 검토, 상태 변경, 공지 작성, 통계 확인 담당

### 3.4 비회원/공개 사용자

- 공지사항과 일부 공개 공급자 탐색 화면을 볼 수 있는 사용자

---

## 4. 전체 화면 구조

화면은 크게 다섯 묶음으로 본다.

1. 공통/인증 화면
2. 요청자 화면
3. 공급자 화면
4. 대화(메시지) 화면
5. 관리자 화면

5. 의뢰 등록
6. 의뢰 상세 확인
7. 견적 비교 및 선택
8. 메시지 상담
9. 연락처 공유 동의

## 5.2 공급자 핵심 흐름

1. 회원가입
2. 로그인
3. 공급자 프로필 작성
4. 검증 서류 제출 및 상태 확인
5. 의뢰 피드 탐색
6. 견적 제출
7. 내 견적 관리

3. 검수 상세 확인
4. 승인 / 보류 / 반려 처리
5. 공지 작성 및 수정
6. 통계 확인

## 6. 화면 상세 기획

## 6.1 공통 / 인증 화면

### 화면 1. 로그인

- 화면명: 로그인
- 사용자: 요청자 / 공급자 / 관리자
- 목적: 이미 가입한 사용자가 서비스에 들어오는 시작점
- 주요 정보:
  - 이메일 입력
  - 비밀번호 입력
- 주요 행동:
  - 로그인
- 기대 결과:
  - 로그인 성공 시 각 사용자 역할에 맞는 화면으로 이동
- 예외/상태:
  - 로그인 실패 시 실패 안내 문구 표시

### 화면 2. 회원가입

- 화면명: 회원가입
- 사용자: 요청자 / 공급자
- 목적: 신규 사용자가 계정을 생성하는 화면
- 주요 정보:
  - 이메일
  - 비밀번호
  - 업체명 또는 사업체 이름
  - 역할 선택(요청자 / 공급자)
- 주요 행동:
  - 회원가입 완료
- 기대 결과:
  - 가입 성공 후 로그인 화면으로 이동
- 예외/상태:
  - 입력 오류 또는 가입 실패 시 오류 안내

### 화면 3. 공지사항 목록

- 화면명: 공지사항 목록
- 사용자: 전체 사용자
- 목적: 운영 공지, 점검, 서비스 안내를 확인하는 화면
- 주요 정보:
  - 공지 제목
  - 요약 문구
  - 게시일
- 주요 행동:
  - 공지 상세 보기
- 기대 결과:
  - 운영 메시지를 누구나 쉽게 확인할 수 있음

### 화면 4. 공지사항 상세

- 화면명: 공지사항 상세
- 사용자: 전체 사용자
- 목적: 특정 공지의 전체 내용을 읽는 화면
- 주요 정보:
  - 제목
  - 본문
  - 게시일
  - 조회수
  - 첨부 파일
- 주요 행동:
  - 첨부 파일 다운로드
  - 목록으로 돌아가기

---

## 6.2 요청자 화면

### 화면 5. 사업자 정보 등록 / 관리

- 화면명: 사업자 정보 등록 / 관리
- 사용자: 요청자
- 목적: 요청자가 의뢰를 등록하기 전에 사업자 정보를 제출하고 승인 상태를 확인하는 화면
- 주요 정보:
  - 상호명
  - 사업자등록번호
  - 담당자 이름 / 연락처 / 이메일
  - 승인 상태(미제출 / 승인 대기 / 승인 완료 / 반려)
- 주요 행동:
  - 최초 제출
  - 반려 또는 대기 상태에서 수정 제출
  - 승인 후 새 의뢰 등록으로 이동
- 상태별 안내:
  - 미제출: 입력 폼 중심
  - 승인 대기: 수정 가능 여부와 대기 안내 강조
  - 승인 완료: 의뢰 등록 가능 안내 강조
  - 반려: 재제출 유도 문구 필요

### 화면 6. 공급자 탐색 목록

- 화면명: 공급자 탐색
- 사용자: 요청자
- 목적: 조건에 맞는 공급자를 찾는 핵심 탐색 화면
- 주요 정보:
  - 회사명
  - 지역
  - 카테고리
  - 월 생산량
  - MOQ
  - OEM/ODM 가능 여부
  - 대표 이미지 또는 로고
- 주요 행동:
  - 키워드 검색
  - 카테고리 / 지역 / 생산량 / MOQ / OEM/ODM 필터 적용
  - 카드 또는 리스트 성격의 목록 탐색
  - 상세 보기 이동
- 화면 설계 포인트:
  - “처음 들어와도 필터를 이해할 수 있는 쉬운 언어”가 중요
  - 검증 상태와 생산 역량은 눈에 잘 띄어야 함

### 화면 7. 공급자 상세

- 화면명: 공급자 상세
- 사용자: 요청자
- 목적: 특정 공급자의 역량과 신뢰 정보를 자세히 확인하는 화면
- 주요 정보:
  - 회사 기본 정보
  - 대표자명
  - 지역
  - 카테고리
  - 월 생산량 / MOQ
  - OEM/ODM 여부
  - 원재료 지원 / 포장 지원 여부
  - 회사 소개
  - 설비 정보
  - 인증서 목록
  - 포트폴리오 이미지
- 주요 행동:
  - 이 공급자에게 의뢰하기
- 상태별 안내:
  - 요청자가 사업자 승인 전이면 즉시 의뢰 불가 안내
  - 승인 후에는 지정 모드 의뢰로 자연스럽게 연결

### 화면 8. 내 의뢰 목록

- 화면명: 내 의뢰 목록
- 사용자: 요청자
- 목적: 내가 올린 의뢰들의 현재 상태를 한눈에 관리하는 화면
- 주요 정보:
  - 의뢰 제목
  - 카테고리
  - 공개/지정 모드
  - 받은 견적 수
  - 상태(작성 중 / 진행 중 / 마감 / 취소)
  - 생성일
- 주요 행동:
  - 새 의뢰 등록
  - 상태별 탭 필터
  - 상세 보기 이동

### 화면 9. 의뢰 등록

- 화면명: 새 의뢰 등록
- 사용자: 요청자
- 목적: 제조 요청 내용을 구조화해서 등록하는 화면
- 주요 정보:
  - 의뢰 제목
  - 카테고리
  - 희망 수량
  - 희망 단가 범위
  - 필요 인증
  - 원재료 제공 조건
  - 포장/라벨링 요구
  - 납기 요구
  - 비고
  - 의뢰 모드(공개 / 지정)
  - 지정 모드일 때 대상 공급자 선택
- 주요 행동:
  - 의뢰 저장 / 게시
- 상태별 안내:
  - 공개 모드: 모든 적합 공급자 대상
  - 지정 모드: 선택 공급자에게만 보임
  - 필수값 누락 시 제출 불가

### 화면 10. 의뢰 상세

- 화면명: 의뢰 상세
- 사용자: 요청자
- 목적: 의뢰 상태와 내용, 후속 액션을 관리하는 중심 화면
- 주요 정보:
  - 의뢰 제목 / 상태
  - 카테고리 / 모드 / 생성일
  - 수량, 단가, 원재료, 포장, 납기, 비고
  - 견적 수
  - 지정 공급자 목록
- 주요 행동:
  - 수정
  - 게시
  - 마감
  - 취소
  - 지정 공급자와 대화 시작
  - 견적 비교 화면 이동
- 상태별 안내:
  - 작성 중(draft): 수정 / 게시 가능
  - 진행 중(open): 수정 / 마감 / 취소 가능
  - 마감(closed) / 취소(cancelled): 읽기 중심

### 화면 11. 견적 비교

- 화면명: 견적 비교
- 사용자: 요청자
- 목적: 받은 견적을 비교하고 선택 또는 거절하는 화면
- 주요 정보:
  - 공급자명
  - 제출일
  - 예상 단가
  - MOQ
  - 납기
  - 샘플비
  - 견적 상태
- 주요 행동:
  - 상세 보기
  - 메시지 이동
  - 견적 선택
  - 견적 거절
- 화면 설계 포인트:
  - 비교가 쉬워야 하므로 표 구조가 적합
  - 선택 시 “다른 견적은 자동 거절” 같은 영향 안내가 필요

## 6.3 공급자 화면

### 화면 12. 공급자 프로필 작성 / 관리

- 화면명: 공급자 프로필 관리
- 사용자: 공급자
- 목적: 공급자의 기본 회사 정보와 제조 역량을 등록하는 화면
- 주요 정보:
  - 회사명
  - 대표자명
  - 연락처
  - 지역
  - 카테고리
  - 설비 요약
  - 월 생산량
  - MOQ
  - OEM/ODM 여부
  - 원재료 지원 여부
  - 포장/라벨링 지원 여부
  - 소개 문구
- 주요 행동:
  - 프로필 저장
  - 검증 제출
- 상태별 안내:
  - 작성 중(draft) / 보완 요청(hold) / 반려(rejected)에서는 수정 범위가 넓음
  - 승인 완료(approved) 이후에는 핵심 연락처 정도만 제한적으로 수정 가능하도록 안내 필요

### 화면 13. 공급자 의뢰 피드

- 화면명: 의뢰 피드
- 사용자: 공급자
- 목적: 응답 가능한 의뢰를 확인하는 화면
- 주요 정보:
  - 의뢰 제목
  - 카테고리
  - 요청 조건
  - 대상 여부 또는 노출 여부
- 주요 행동:
  - 상세 보기
  - 견적 작성으로 이동

### 화면 14. 공급자 의뢰 상세

- 화면명: 공급자용 의뢰 상세
- 사용자: 공급자
- 목적: 특정 의뢰에 대한 상세 조건을 보고 견적 작성 여부를 판단하는 화면
- 주요 정보:
  - 요청 내용 전체
  - 수량 / 단가 범위 / 납기 / 포장 요구 / 인증 요구
- 주요 행동:
  - 견적 작성
  - 대화 이동

### 화면 15. 견적 제출

- 화면명: 견적 제출
- 사용자: 공급자
- 목적: 요청자에게 실제 견적을 보내는 화면
- 주요 정보:
  - 예상 단가
  - MOQ
  - 납기
  - 샘플 비용
  - 비고
- 주요 행동:
  - 견적 내용 입력
  - 제출 전 확인
  - 최종 제출
- 화면 설계 포인트:
  - 제출 전에 요약 확인 단계가 있으면 실수를 줄일 수 있음

### 화면 16. 내 견적 관리

- 화면명: 내 견적 관리
- 사용자: 공급자
- 목적: 제출한 견적을 상태별로 관리하는 화면
- 주요 정보:
  - 의뢰명
  - 카테고리
  - 제출일
  - 단가 / MOQ / 납기
  - 견적 상태
- 주요 행동:
  - 수정
  - 철회
  - 메시지 이동
- 상태별 안내:
  - 제출 완료(submitted) 상태에서만 수정 / 철회 가능
  - 종료된 견적은 읽기 중심으로 유지

## 6.4 메시지 / 상담 화면

### 화면 17. 메시지 목록

- 화면명: 메시지 목록
- 사용자: 요청자 / 공급자
- 목적: 진행 중인 상담을 한눈에 확인하는 화면
- 주요 정보:
  - 상대 이름
  - 상대 역할
  - 관련 의뢰 제목
  - 마지막 메시지
  - 읽지 않은 메시지 수
  - 마지막 활동 시간
- 주요 행동:
  - 대화 상세 진입
- 화면 설계 포인트:
  - 읽지 않은 대화는 시각적으로 더 눈에 띄어야 함

### 화면 18. 대화 상세

- 화면명: 대화 상세
- 사용자: 요청자 / 공급자
- 목적: 의뢰/견적 맥락 안에서 상담을 이어가는 화면
- 주요 정보:
  - 상대 정보
  - 관련 의뢰명
  - 메시지 목록
  - 첨부 파일
  - 읽음 처리
  - 연락처 공유 상태
- 주요 행동:
  - 메시지 작성
  - 파일 첨부
  - 연락처 공유 요청
  - 연락처 공유 승인
  - 요청 철회
- 상태별 안내:
  - 연락처는 상호 승인 전까지 숨김
  - 상호 승인 후에만 양측 연락처 카드 노출
  - 읽음/미읽음은 자동 반영

## 6.5 관리자 화면

### 화면 19. 검수 큐

- 화면명: 검수 큐
- 사용자: 관리자
- 목적: 공급자 제출 건을 상태별로 검토하는 작업 목록 화면
- 주요 정보:
  - 회사명
  - 현재 상태
  - 제출일
  - 대기일수
- 주요 행동:
  - 상태 필터
  - 기간 필터
  - 정렬
  - 상세 진입
- 화면 설계 포인트:
  - 관리자는 빠르게 우선순위를 판단해야 하므로 목록 가독성이 중요

### 화면 20. 검수 상세

- 화면명: 검수 상세
- 사용자: 관리자
- 목적: 제출 자료를 보고 승인 / 보류 / 반려 결정을 내리는 화면
- 주요 정보:
  - 회사 기본 정보
  - 제출 파일 목록
  - 검수 이력
  - 내부 메모 / 사용자 공개 메모 / 사유 코드
- 주요 행동:
  - 승인
  - 보류
  - 반려
  - 파일 다운로드
- 상태별 안내:
  - `hold`는 보완 요청, `reject`는 현재 제출 건 반려 의미가 명확해야 함
  - 이후 재제출이 가능하다는 운영 규칙을 문구로 안내하는 것이 좋음

### 화면 21. 공지사항 관리

- 화면명: 공지사항 관리
- 사용자: 관리자
- 목적: 서비스 운영 공지를 작성하고 관리하는 화면
- 주요 정보:
  - 제목
  - 상태(작성중 / 게시됨 / 보관됨)
  - 작성자
  - 게시일
  - 조회수
- 주요 행동:
  - 새 공지 작성
  - 편집
  - 게시
  - 보관
- 화면 설계 포인트:
  - 운영자는 공지 작성과 상태 변경을 빠르게 할 수 있어야 함

### 화면 22. 통계 대시보드

- 화면명: 통계 대시보드
- 사용자: 관리자
- 목적: 운영 현황을 한눈에 파악하는 화면
- 주요 정보:
  - 전체 사용자 수
  - 요청자 / 공급자 / 관리자 수
  - 공급자 상태별 현황
  - 검수 대기 현황
  - 의뢰 현황
  - 기간 필터
- 주요 행동:
  - 기간 변경
  - 현황 확인
- 화면 설계 포인트:
  - 숫자만 나열하기보다 카드와 간단한 막대형 시각화가 이해하기 쉽다.

## 7. PRD 기준 확인이 필요한 화면

PRD 기준으로 보면 아래 항목은 별도 관리 화면 여부를 한 번 더 확인할 필요가 있다.

### 확인 필요 1. 관리자 회원 목록

- PRD에는 “관리자는 사용자/공급자 목록을 조회할 수 있다”가 포함되어 있다.
- 현재 구현 라우트 기준으로는 검수 큐, 공지, 통계는 보이지만 `회원 목록 전용 화면`은 분리되어 보이지 않는다.
- 따라서 이 항목은 아래 둘 중 하나로 정리해야 한다.
  - 이미 다른 화면 안에서 해결된 요구인지 확인
  - 아니면 별도 관리자 화면으로 다음 단계 과업 또는 미완료 항목으로 다시 정리

## 8. 화면 기획 시 공통 원칙

- 어려운 용어 대신 업무 언어로 쓴다.
- 상태값은 사용자가 이해할 수 있는 표현으로 바꿔 보여준다.
- 중요한 행동은 한 화면에서 한 번에 이해할 수 있어야 한다.
- 오류 문구는 “왜 안 되는지”와 “어떻게 해야 되는지”를 같이 알려준다.
- 모바일에서도 핵심 흐름은 가능해야 하므로 입력 폼과 표 구조는 지나치게 복잡하지 않아야 한다.

---

## 9. 한 줄 정리

이 MVP의 화면 구조는 `공급자 탐색 -> 의뢰 등록 -> 견적 비교 -> 메시지 상담 -> 필요 시 연락처 공유`라는 하나의 신뢰형 매칭 흐름을 중심으로 설계되어야 하며, 관리자 화면은 이 흐름을 뒷받침하는 `검수 / 공지 / 통계` 운영 도구 역할을 한다.
