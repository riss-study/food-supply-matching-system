## 1. 이 문서의 역할

이 문서는 `.sisyphus` 내부 기획 문서의 기준점을 다시 세우기 위한 **마스터 작업 계획서**다.

목표는 세 가지다.

1. 어떤 문서가 현재 기준 문서인지 명확히 정한다.
2. 장기 비전과 MVP 범위를 분리해 문서 간 충돌을 없앤다.
3. 이후 문서를 수정할 때 어디까지 연쇄 반영해야 하는지 규칙을 고정한다.

## 2. 현재 문서 진단 요약

현재 문서 세트의 핵심 문제는 기능 부족이 아니라 **기획 기준선의 혼선**이다.

- 장기 비전은 `풀 패키지 플랫폼`인데, 상세 PRD는 `핵심 매칭 MVP`를 기준으로 쓰여 있다.
- 질문지 답변은 조건부이거나 보류인데, 후속 문서는 이를 `확정`으로 과하게 해석한 부분이 있다.
- 일정, 인증 정책, 지도/채팅/알림/리뷰 관련 가정이 문서마다 다르다.
- 요약 문서와 PRD가 원문보다 더 강한 약속을 해버려서, 읽는 사람이 무엇을 실제로 만들어야 하는지 헷갈린다.

즉, 지금 필요한 것은 새 아이디어가 아니라 **문서 권위 체계 재정렬 + MVP 범위 재선언 + 후속 문서 동기화**다.

## 4. 수정된 제품 기준선

### 4.1 제품 비전과 MVP를 분리한다

- **장기 비전**: 식품 제조 매칭 + 신뢰 관리 + 운영 지원 + 콘텐츠/커뮤니티를 포함하는 풀 패키지 플랫폼
- **Phase 1 / MVP**: 검증 가능한 공급자 프로필, 의뢰 등록, 견적 흐름, 메시지 기반 커뮤니케이션, 관리자 검수 기능까지 포함한 **신뢰형 매칭 코어**
- **Phase 2+**: 커뮤니티, 교육/컨설팅, 리뷰 고도화, 지도/외부 API, AI 추천, OCR, 실시간성 강화

이 프로젝트의 문제는 비전이 큰 것이 아니라, 비전이 그대로 MVP 범위처럼 서술된 점이다. 앞으로는 모든 문서에서 이 구분을 유지한다.

### 4.2 MVP에서 확정할 기준

- 요청자와 공급자 모두 사업자 기반의 B2B 사용자를 우선 대상으로 둔다.
- 공급자는 가입 자체는 가능하지만, **검증 상태에 따라 노출/신뢰/기능 권한을 분리**한다.
- HACCP은 플랫폼에서 중요하게 다루지만, MVP의 기본 구현은 **수동 검수 + 표시 체계**다.
- 리뷰, 지도, WebSocket, OCR, 외부 검증 API는 MVP 필수 경로에서 뺀다.

### 4.3 MVP에서 제거하거나 낮춰야 할 과한 약속

- `모든 기능 완성 후 런칭`
- `실시간 채팅/WebSocket 필수`
- `리뷰 시스템 없이 리뷰 정렬/필터 제공`
- `지도 뷰를 MVP 핵심 기능으로 전제`
- `외부 API 연동이 없으면 인증 기능 자체가 성립하지 않는 구조`

## 5. 핵심 의사결정 정리

### 5.1 사용자 및 인증

- 요청자: 사업자 기반 브랜드/운영 조직 우선
- 공급자: 가입 가능, 검증 완료 시 검색 노출/신뢰 배지/견적 참여 권한 강화
- 인증 방식: MVP는 수동 심사 중심
- 인증 종류: 사업자 등록 정보, HACCP 및 기타 식품 관련 인증 정보 표시

### 5.2 탐색과 검색

- MVP 필수 뷰: 카드, 리스트
- MVP 필수 필터: 카테고리, 지역, 인증 상태, MOQ, OEM/ODM, 생산 역량
- 지도 뷰: 별도 검토 항목으로 관리
- 리뷰 기반 정렬/필터: 리뷰 데이터가 생기는 단계 이후로 이관

### 5.3 커뮤니케이션

- MVP는 `실시간 채팅`보다 `메시지 스레드` 개념이 더 정확하다.
- 메시지, 파일 첨부, 읽음 상태, 연락처 공유 동의 흐름까지를 1차 범위로 둔다.
- 실시간 푸시/WebSocket/타이핑 인디케이터는 Phase 2 이후 검토한다.

### 5.4 단계 계획

- Phase 1: 신뢰형 매칭 코어
- Phase 2: 사용자 경험 고도화 + 운영 보조
- Phase 3: AI/외부 연동/확장 기능

## 6. 실제 작업 순서

### Step 1. 기준 문서 정리

목표: 더 이상 초안/요약/PRD가 서로 다른 말을 하지 않도록 한다.

- `1st-phase-questionnaire-v2.md`에 기록 문서라는 성격을 명시
- `1st-phase-PRD-v0.1.md`를 Archive로 명시
- `1st-phase-requirements-final.md`를 한국어 기준 요구사항 문서로 재정렬
- `requirements-final-summary-v1.1.md`를 기준 문서의 축약본으로 정리

### Step 2. 상세 PRD 동기화

목표: 구현 기준 문서가 기준 요구사항과 충돌하지 않도록 맞춘다.

- `PRD-v1.0-MVP-Korean.md` 수정
- `PRD-v1.0-MVP-English.md`를 한글 PRD 기준으로 동기화
- `PRD-v1.0-MVP.md`를 혼합 참고본으로 정리하거나 핵심 섹션만 일치시킴

### Step 3. 실행 준비 범위 확정

목표: 개발 시작 전에 필요한 설계 산출물을 분리한다.

- 정보 구조/화면 흐름
- 도메인 모델 및 권한 정책
- 데이터 모델 초안
- API 초안
- 관리자 검수 플로우
- 테스트 기준 및 수용 조건

### Step 4. 개발 착수 계획 작성

목표: 구현 전에 작업 단위를 실제 개발 가능한 수준으로 쪼갠다.

- 프론트엔드 워크스트림
- 백엔드 워크스트림
- 검수/테스트 워크스트림
- 문서/정책 워크스트림

### Step 5. 문서 정합성 검증

목표: 수정한 문서가 실제로 같은 범위를 말하는지 검증한다.

- 기준 문서와 요약 문서 대조
- 한글 PRD와 영문 PRD 대조
- MVP 포함/제외 항목 대조
- 검색/인증/메시지 정책 교차 점검

## 8. 문서 변경 전파 규칙

앞으로 특정 항목을 수정하면 아래 순서로 반드시 연쇄 반영한다.

| 변경 대상 | 반드시 함께 수정할 문서 |
|-----------|--------------------------|
| 사용자 역할/인증 정책 | 요구사항 문서, 요약 문서, 한글/영문 PRD |
| MVP 범위 | 요구사항 문서, 요약 문서, 모든 PRD, 이 계획서 |
| 기술 스택 | 요구사항 문서, 모든 PRD, 향후 아키텍처 문서 |
| 검색/정렬/지도 정책 | 요구사항 문서, 요약 문서, PRD 검색 섹션 |
| 메시지/알림 방식 | 요구사항 문서, 요약 문서, PRD 커뮤니케이션 섹션 |

1. 질문지 원문은 기록으로 남기되, 후속 확정 문서가 이를 어떻게 해석했는지 분리해서 쓴다.
2. 요약 문서는 기준 문서를 압축만 해야지, 새로운 약속을 추가하면 안 된다.
3. 영문 문서는 한글 기준 문서 수정 후 마지막에 동기화한다.

## 10. 완료 조건

이번 문서 정리 작업은 아래 조건이 모두 충족되면 완료로 본다.

## 11. 이번 문서 정리에서 적용할 편집 원칙

- 새로운 아이디어 추가보다 기존 충돌 제거를 우선한다.
- 사용자 답변의 의도는 살리되, 과한 해석은 제거한다.
- 구현 불확실성이 큰 항목은 `후속 검토`로 내리고, 현재 해야 할 일만 MVP에 남긴다.
- 동일한 결정을 여러 문서에 쓸 때 표현 강도를 통일한다.
